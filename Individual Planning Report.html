1) Data Description

This project utilizes two datasets provided as CSV files: players.csv and sessions.csv. The data was collected from a Minecraft research server run by a Computer Science group at UBC, where players.csv contains information on individual players, while sessions.csv contains information on individual play sessions.

Dataset Summaries

players.csv

  • 196 observations, each representing a unique player
  • 7 columns/variables
  • Variable Details:
    1. experience (character): A categorical variable describing the player's experience (e.g., 'Pro', 'Veteran', 'Amateur', 'Regular').
    2. subscribe (boolean/logical vector): A boolean variable (True/False) indicating if the player is subscribed to a game-related newsletter.
    3. hashedEmail (character): A categorical variable serving as a unique identifier for each player.
    4. played_hours (double): A quantitative variable for the total number of hours played.
    5. name (character): A categorical variable for the player's in-game name.
    6. gender (character): A categorical variable for the player's gender.
    7. Age (double): A quantitative variable for the player's age.

sessions.csv

  • 1,535 observations, each representing one play session
  • 5 columns/variables
  • Variable Details:
    1. hashedEmail (character): A categorical variable serving as a unique identifier for each player.
    2. start_time (character): The start time of each session as a categorical variable, stored in plain text.
    3. end_time (character): The end time of each session as a categorical variable, stored in plain text.
    4. original_start_time (double): The start time of each session as a quantitative variable, stored as a Unix timestamp.
    5. original_end_time (double): The end time of each session as a quantitative variable, stored as a Unix timestamp.

Issues within the Data

Upon initial analysis of the data, there are a few potential errors that will need to be addressed.

  • players.csv's Age column contains 2 missing/NA values, and sessions.csv's end_time and original_end_time columns each have 2 missing/NA values. Because these errors are fairly minor, we can choose to remove them or perform mean imputation.
  • Additionally, start_time and end_time would likely need to be converted to a numerical form for any time-based analysis, but we are already given their Unix 'original' counterparts.
  • Finally, and most glaringly, some data is highly skewed. played_hours, for example, has a mean of 5.85 yet a median of 0.10, implying that most players have very few hours while a few players have very many hours. This trend continues into the experience variable as well, where there are significantly more Amateurs (63) than Pros (14).

2) Questions

Broad Question: What player characteristics and behaviours are most predictive of subscribing to a game-related newsletter?

Specific Question: Can a player's experience level, total hours played, age, and gender be used to predict whether they will subscribe to the game-related newsletter in the dataset?

The players.csv dataset is well suited to answer this question because it gives us all relevant information, including the response/explanatory variables. It's important to note that this is a binary classification problem, and sessions.csv is not immediately required for this specific question.

4) Methods and Plan

Proposed Method: K-NN Classification

Q: Why is this method appropriate?

Our problem is a binary classification task, as we are only predicting true or false. Given our small number of predictors and strong dataset size, K-NN is computationally feasible, and requires few assumptions about the underlying data distribution.

Q: Which assumptions are required, if any, to apply the method selected?

K-NN is highly sensitive to the scale of features. Because distance is calculated using variables like Age (range ~9-58) and played_hours (range ~0-223), an unstandardized analysis of this data would make the played_hours variable dominate the distance calculation. Thus, we must normalize/scale all quantitative features to a similar range.

Q: What are the potential limitations or weaknesses of the method selected?

We must choose an optimal K value, which can be tricky. A poor choice of K can lead to overfitting or underfitting, resulting in a weaker model. Another weakness arises when considering the opportunity cost of using K-NN: it doesn't produce a simple "model" (like regression coefficients), making it slightly less interpretable in terms of which features are most important.

Q: How are you going to compare and select the model?

We will tune K using 10-fold cross-validation on the training set. After testing on a range of K values (starting off with a range akin to 1-30), we can determine the strongest model by picking the K value with highest mean cross-validation accuracy.

Q: How are you going to process the data to apply the model? For example: Are you splitting the data? How? How many splits? What proportions will you use for the splits? At what stage will you split? Will there be a validation set? Will you use cross validation?

First, split the players.csv data into a training set (80% of data) and a test set (20% of data) using initial_split. Now that the data is split, we can define a recipe, and convert subscribe to a factor (it is the response variable!). Scale and center our data to prevent variables from having more weight in the voting process of K-NN.

Next, define a K-NN model specification using nearest_neighbor(), setting mode = "classification" and neighbors = tune() to mark K for tuning. Combine the recipe and the model into a workflow, then begin tuning with tune_grid() and 10 cross-validation folds of the training data. After finding the best K, test the model's accuracy on the unseen test set, and report the accuracy, precision, recall, and a confusion matrix for a full picture of its performance.
